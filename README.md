# Quant Team Framework — Autonomous Quant Research Loop

A small, from-scratch Python project that turns a trading idea into researched,
implemented, reviewed and backtested strategy candidates.

The project intentionally mirrors a simple four-agent experiment:

1. **Research agent** queries Semantic Scholar first, falls back to academic
   domains, and proposes a signal using only verified citations.
2. **Implementation agent** writes a `generate_signals(data)` Python function.
3. **Verification agent** reviews the code and sends material issues back for
   one qualitative revision. Reproducible safety failures can use all three
   revision rounds.
4. **Backtest agent** computes performance deterministically, without an LLM.

The loop runs independently several times (10 by default) and ranks valid
strategies by Sharpe ratio.

## Architecture

```text
research question in plain English
                 |
                 v
Research -> Implementation <-> Verification -> Backtest
   ^                                             |
   |------------- next independent run ----------|
                                                 v
                                      rank candidates by Sharpe
```

Market data is selected and prepared **before** the agents run. It is never
generated by an LLM and is cached locally.

The verifier combines an LLM review with deterministic checks: allowed
imports, executable interface, finite output shape, explicit future-looking
patterns, and prefix invariance (past signals must not change when future rows
are removed). Objective checks run before the reviewer to avoid wasting an LLM
call on broken code. Current-close signals are valid because execution occurs
the next day. Candidates with unresolved objective failures after three
revisions are excluded. LLM-only concerns get one correction attempt and are
then retained as advisories if every reproducible safety check passes.

## Installation

Requires Python 3.10+.

```bash
git clone <repository-url>
cd Quant-team-framework
python -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt
cp .env.example .env
```

Set your key in `.env`:

```dotenv
OPENAI_API_KEY=sk-...
# OPENAI_MODEL=gpt-4o
```

`.env` and generated research artifacts are excluded from Git.

## Interactive usage

```bash
python main.py
```

The terminal asks for:

- the research question in plain English (for example, "pairs trading on
  the S&P 500");
- the number of independent candidates (10 by default);
- backtest start and end dates in `YYYY-MM-DD` format.

Everything else is fixed before the agents start: S&P 500 close prices are
downloaded from Yahoo Finance and cached locally. The agents never fetch or
generate market data themselves.

## Advanced CLI options

The defaults (S&P 500, Yahoo close prices) can be overridden with flags —
run `python main.py --help` for the full list.

Available curated sectors are `banking`, `technology`, `energy`,
`healthcare`, and `consumer`.

Available index universes are `sp500`, `nasdaq100`, `cac40`, `dax40`,
`ftse100`, and `nikkei225`. Their current constituents are resolved before
the data download and converted to Yahoo Finance symbols automatically.

A `price_volume` model type adds volume data alongside close prices.

### Universe Agent

Pass `--universe agent` for a broad request such as “crypto universe” or
“liquid European equities.” In interactive mode, the agent asks up to four
English clarification questions before returning concrete Yahoo Finance
tickers.

```bash
python main.py --universe agent
# Describe the asset universe: liquid crypto assets
```

For automation, omitted answers use the agent's defaults:

```bash
python main.py --non-interactive \
  --universe agent \
  --universe-request "top liquid crypto assets excluding stablecoins" \
  --idea "cross-sectional crypto momentum" \
  --runs 3
```

Specific generated questions can also be answered with repeatable
`--universe-answer key=value` arguments.

For example:

```bash
python main.py --non-interactive \
  --universe cac40 \
  --idea "cross-sectional momentum on CAC 40 stocks" \
  --runs 3
```

## Automated CLI usage

Quick pairs-trading experiment:

```bash
python main.py \
  --non-interactive \
  --source yahoo \
  --universe custom \
  --tickers JPM,BAC,WFC,C \
  --model-type price \
  --idea "pairs trading on S&P 500 banks" \
  --runs 10 \
  --start 2018-01-01 \
  --end 2024-12-31
```

Price-and-volume experiment over the technology universe:

```bash
python main.py \
  --non-interactive \
  --source yahoo \
  --universe technology \
  --model-type price_volume \
  --idea "cross-sectional momentum confirmed by unusual volume" \
  --runs 10
```

## Custom CSV datasets

CSV files use long format:

```csv
date,ticker,close,volume
2024-01-02,JPM,170.10,10800000
2024-01-02,BAC,33.55,42100000
```

`date`, `ticker`, and `close` are required. `volume` is required only for a
`price_volume` model. See `examples/market_data.csv`.

```bash
python main.py \
  --non-interactive \
  --source csv \
  --csv-path examples/market_data.csv \
  --universe custom \
  --tickers JPM,BAC \
  --model-type price_volume \
  --runs 3
```

## Outputs

- `best_strategy.py`: generated code with the highest Sharpe;
- `best_strategy_report.json`: research summary and metrics for the winner;
- `research_runs_report.json`: ranked valid runs and all failed candidates.

A failed verification or backtest invalidates only that candidate. There is no
hidden fallback strategy: every ranked strategy was produced by the
implementation agent and accepted by the verification agent.

## Tests

Tests do not call OpenAI or Yahoo Finance:

```bash
pip install -r requirements-dev.txt
pytest
```

## Important limitation

This is deliberately a basic learning project, not an investment system.
Candidates are generated, selected and ranked on the same historical sample.
There is no walk-forward or out-of-sample evaluation, so the reported Sharpe
is strongly exposed to selection bias and overfitting. Do not use the output
to trade real capital.

Current index constituents also introduce survivorship bias when testing old
periods. Generated strategy code is executed in-process after verification;
a production system must isolate it in a restricted subprocess or container.
